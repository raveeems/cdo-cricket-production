==================================================

BUSINESS GOAL
Some users may be busy for 2–3 days and may not be able to react before every match. I want to allow them to still remain active by selecting backup players in advance.

This backup feature is only for users who want protection if one or more of their selected Playing XI players do not make the official Playing XI.

IMPORTANT PRODUCT RULE
User can select backups only BEFORE official Playing XI is announced / before entry lock.
Once admin sets official Playing XI (or once entry lock condition is reached), backups cannot be edited.

SCOPE
This feature is for MAIN PLAYING XI only.
Do NOT redesign the whole game.
Do NOT change pot logic automatically.
Do NOT link this directly to weekly penalty logic.
This is only a team-protection feature.

BACKUP MODEL TO IMPLEMENT
Use the simpler and safer model:

- User selects main team as normal
- User can additionally select up to 2 Playing XI backup players
- These 2 backups are global priority backups, not mapped player-to-player backups

Meaning:
- Backup 1 = first replacement priority
- Backup 2 = second replacement priority

NOT this:
- Raja -> Dhoni
- David -> Samson

I do NOT want per-player replacement mapping.

HOW REPLACEMENT SHOULD WORK
At resolution time, after official Playing XI is known:

1. Find selected main-team players who are NOT in the official Playing XI
2. Use Backup 1 first if it is valid and in official Playing XI
3. Use Backup 2 next if needed and valid
4. Maximum 2 replacements only
5. If backups are also not in official XI, no replacement for those slots
6. If no selected main player misses, backups are ignored

- Backup player cannot already be in main XI
- Backup player cannot duplicate another backup
- Backup player cannot overlap with main Impact player or Impact backup(s)
- Backups can be edited only before official Playing XI announcement / entry lock
- Maximum 2 XI backups only
- If 3 or more selected players miss, only first 2 valid replacements happen
- Do NOT auto-transfer captain/vice-captain if captain or vice-captain misses
- Backup replaces only the player slot, not captaincy or vice-captaincy

FEATURE 2 — POT MANAGEMENT PENALTY MODE
==================================================

BUSINESS GOAL
Some users participate irregularly. I want a way for admin to manually penalize users in pot calculation when needed, without the app trying to fully automate weekly attendance detection.

IMPORTANT PRODUCT RULE
The app does NOT need to automatically detect:
- how many matches a user missed in the week
- whether the user deserves penalty
- weekly attendance policy

Admin will track that externally/manually.

The app only needs to support a structured manual penalty mechanism in Pot Management.

REQUIRED POT MANAGEMENT MODES

MODE 1
As per contest entries
- current behavior
- only users who entered that contest count toward pot

MODE 2
Contest entries + penalty users
- all actual contest entrants count normally
- admin can additionally select users to add a fixed penalty contribution
- these penalty users affect POT ONLY
- they do NOT become contest participants
- they do NOT appear in leaderboard/standings for that match
- they do NOT become eligible for winnings for that match

IMPORTANT
Penalty users are financial contributors only.
They are NOT participants.

UI REQUIREMENT
In Pot Management, add:

1. Calculation mode selector
- As per contest entries
- Contest entries + penalty users

2. If penalty mode selected:
- show penalty amount per user (default 30, editable if needed)
- show list of users for admin to select penalty users

3. Very important:
Users already counted via contest entry should NOT be selectable as penalty users
Best behavior:
- disable them in list
- show note like “Already included via contest entry”

4. Show clear preview
Example:
- Contest entries: 7 × 30 = 210
- Penalty users: 2 × 30 = 60
- Total pot = 270

This preview is mandatory.

DATA MODEL REQUIREMENT
Store enough information for auditability.

Preferred approach:
Create a linked structure/table for penalty users rather than just raw JSON if current architecture supports that cleanly.

Need to store:
- calculation_mode
- penalty_amount_per_user
- penalty user IDs
- who saved it / when, if current architecture already supports audit-like fields

FORMULA
Mode 1:
pot = contest_entry_count × per_user_amount

Mode 2:
pot = (contest_entry_count × per_user_amount) + (penalty_user_count × penalty_amount)

IMPORTANT
Penalty users should not affect:
- match participants list
- leaderboard
- prize eligibility

They affect pot total only.
