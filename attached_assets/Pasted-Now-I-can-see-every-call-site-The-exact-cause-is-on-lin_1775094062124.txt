**The exact bug is confirmed now — it is on line 1020 itself, but only under one condition:**

`isCurrentlyImpact` is captured from the render-time closure. If `impactPlayerIds` in the closure at the time `setPlayerImpact` was called is **stale** — i.e., the player was already removed from a previous tap that hasn't re-rendered yet — then `isCurrentlyImpact = false`, `!isCurrentlyImpact = true`, and `setMatchPlayers` sets `isImpactPlayer: true` on the player.

**This is the exact bug.** A double-tap, or two rapid taps on the same chip, causes `isCurrentlyImpact` to be `false` on the second tap (because the first tap already removed it from the set in a pending state update), so `setMatchPlayers` restores `isImpactPlayer: true`.
